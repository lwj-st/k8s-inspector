# K8s故障自动匹配系统设计文档

## 1. 文档目标

本文档用于定义一套面向实施运维人员的单集群 K8s 故障排查系统。系统目标是通过只读采集、规则匹配、证据展示和补充分析，帮助运维人员快速定位故障原因并给出处置建议。

本文档聚焦首版可落地方案，优先保证轻量、可控、易维护，不追求大而全的可观测平台能力。

## 2. 产品定位

- 产品形态：独立部署的轻量化排障系统，包含前端页面和后端服务
- 使用对象：实施运维、平台运维
- 部署范围：单集群
- 权限模型：仅保留一个管理员角色
- 操作边界：系统只做排查、展示、分析和建议输出，不执行任何集群改动

### 2.1 设计原则

- 只读优先：仅通过 Kubernetes 只读接口采集数据，不做集群写操作
- 规则优先：故障判断以预录入模板为主，大模型只做补充分析
- 证据优先：所有结论都应尽量附带状态、事件、日志或资源数据
- 轻量优先：服务数量尽量少，首版控制在少量 Pod 内完成部署
- 可迁移优先：模板、白名单、排查记录支持导出为 JSON/YAML

## 3. 范围定义

### 3.1 本期纳入范围

本期系统需要覆盖以下三类场景：

1. 集群自检
2. 命名空间巡检
3. 定点排查

### 3.2 本期基座服务排查范围

根据现有部署脚本 `01-17`，本期纳入自动排查的 K8s 基座服务范围如下：

- Kubernetes 控制面初始化状态
- `containerd`
- `kubelet`
- `Calico CNI`
- `ingress-nginx`
- `NVIDIA device plugin`
- `Ascend device plugin`

### 3.3 本期明确不纳入范围

- `08-NFS` 不纳入 K8s 基座服务自动排查范围，仅作为可选存储前置能力
- 多集群统一纳管
- 自动修复、自动执行命令、自动变更集群
- 面向业务方的复杂权限体系

## 4. 核心业务场景

### 4.1 集群自检

系统自动或定时检查单集群基础状态，识别控制面和基座服务的异常。

输出内容包括：

- 当前异常对象
- 异常状态
- 关联事件或日志摘要
- 初步故障原因
- 处理建议

### 4.2 命名空间巡检

用户手动输入命名空间后，系统检查该命名空间下指定服务或全部目标对象的健康情况。

巡检对象包括但不限于：

- `Pod`
- `Service`
- `Ingress`
- `TLS Secret`
- `DaemonSet`

规则如下：

- 这些对象以状态排查为主
- 如果某类对象不存在，则直接跳过，不视为异常
- `Pod` 是巡检核心对象，需要补充日志、事件和资源证据

### 4.3 定点排查

用户已知某类故障现象时，手动触发定点排查。系统基于已录入模板执行多条件匹配，输出匹配到的故障原因和处理建议。

如果模板命中明确原因，系统需要输出：

- 故障原因
- 命中证据
- 预录入处理建议
- 预录入处理命令
- 执行提示和风险说明

如果模板未命中，则仅输出：

- 已采集问题信息
- 证据摘要
- 通用处理建议

## 5. 功能设计

### 5.1 集群自检

集群自检负责检查基础组件状态，建议支持定时执行和手动执行两种方式。

检查内容包括：

- Node 状态是否正常
- 控制面组件状态是否正常
- `containerd` 运行状态
- `kubelet` 运行状态
- `Calico` 组件状态
- `ingress-nginx` 控制器状态
- `NVIDIA/Ascend device plugin` 状态

输出要求：

- 展示异常对象、所属命名空间、所属节点、状态摘要
- 对非正常 Pod 展示 `describe` 摘要
- 对运行期异常附加日志摘要

### 5.2 命名空间巡检

用户输入命名空间后，系统需按下列维度巡检：

- Pod 状态
- Pod 重启次数
- Pod 最近事件
- Pod 日志异常关键字
- Pod CPU/内存使用情况
- Service 状态
- Ingress 状态
- TLS Secret 存在性与有效期状态
- DaemonSet 状态

对象处理规则：

- `Service`、`Ingress`、`TLS Secret`、`DaemonSet` 以状态排查为主
- 对象不存在时跳过，不报异常
- 对象存在但状态异常时，输出状态与关联信息

Pod 证据规则：

- 若 `Pod` 非正常状态，必须输出对应 `describe` 摘要
- 若 `Pod` 为 `CrashLoopBackOff`、`Error`、`OOMKilled` 等运行期异常，必须输出日志摘要
- 若 `Pod` 为 `Pending`、`ImagePullBackOff`、`CreateContainerConfigError` 等调度或配置类异常，优先输出 `describe` 和事件信息

### 5.3 定点排查

定点排查依赖故障模板库，按用户选择的排查方向执行规则匹配。

#### 5.3.1 匹配维度

- 日志关键字
- Pod 状态
- 容器重启次数
- 事件关键字
- CPU/内存使用率
- 节点状态
- 关联对象状态

#### 5.3.2 多 Pod 联合匹配

系统需要支持多 Pod 联合命中作为故障判定条件。

例如：

- 同一 `label selector` 下至少 `N` 个 Pod 同时命中某日志关键字
- 不同 `label selector` 对应的 Pod 组分别命中不同关键字后共同判定故障

因此模板规则需要支持以下表达能力：

- 匹配对象组
- 对象组的 `namespace + label selector`
- 关键字命中条件
- 命中数量阈值
- 逻辑关系，如 `AND/OR`

### 5.4 故障模板管理

系统需要提供前端页面管理故障模板，至少支持：

- 新增
- 编辑
- 启用
- 停用
- 删除
- 导入
- 导出

模板字段建议包括：

- 模板名称
- 适用场景
- 对象范围
- 命名空间范围
- `label selector`
- 匹配条件
- 多 Pod 联合规则
- 故障原因
- 处理建议
- 处理命令
- 风险提示
- 启用状态

### 5.5 白名单管理

系统需要支持手动白名单能力，用于过滤已知无害日志关键字。

白名单生效维度定义为：

- `namespace + label selector + keyword`

白名单规则：

- 白名单由管理员手动添加
- 生效后，对应范围内不再将该关键字作为异常日志参与巡检与匹配
- 白名单仅影响关键字过滤，不影响 Pod 状态、事件、资源等其他检查项

### 5.6 大模型辅助分析

系统支持配置大模型，默认适配 `Qwen`。

触发规则：

- 如果已配置可用 `API Key`，在规则未命中或命中不足时默认触发
- 如果未配置大模型，则仅输出规则排查结果

输出边界：

- 大模型仅输出补充分析建议
- 不执行任何集群操作
- 不向集群下发命令
- 即使展示命令，也仅作为人工执行参考

## 6. 页面设计

首版前端页面至少包含以下模块：

1. 集群总览 / 自检结果页
2. 命名空间巡检页
3. 定点排查页
4. 故障模板管理页
5. 白名单管理页
6. 系统配置页

### 6.1 集群总览 / 自检结果页

用于展示：

- 当前集群总体健康状态
- 异常组件列表
- 最近自检结果
- 高优先级问题摘要

### 6.2 命名空间巡检页

用于：

- 输入命名空间
- 可选输入 `label selector`
- 手动触发巡检
- 查看对象状态、异常证据和建议

### 6.3 定点排查页

用于：

- 选择排查方向
- 输入命名空间和对象范围
- 触发规则匹配
- 查看匹配度、证据、原因、建议和预录入命令

### 6.4 故障模板管理页

用于：

- 管理模板库
- 编辑多 Pod 关联规则
- 管理建议与命令
- 导入导出 JSON/YAML

### 6.5 白名单管理页

用于：

- 按 `namespace + label selector + keyword` 维护白名单
- 查看命中范围
- 启停白名单

### 6.6 系统配置页

用于：

- 配置大模型参数
- 配置默认巡检策略
- 查看本地数据库导入导出信息

## 7. 数据采集设计

### 7.1 数据来源

- Kubernetes API
- Metrics API
- Pod 日志
- Event

### 7.2 采集原则

- 全部采用只读权限
- 不直接进入容器执行命令
- 不修改任何 K8s 资源

### 7.3 主要采集项

- Cluster / Node / Namespace 基础信息
- Pod 状态、重启次数、节点归属
- Pod `describe` 关键信息
- Pod 当前日志与必要时的前一次日志
- Event 列表
- CPU / 内存指标
- Service / Ingress / TLS Secret / DaemonSet 状态

## 8. 数据存储设计

系统采用本地数据库存储，首版优先选择轻量化方案。

建议：

- 默认使用 `SQLite`
- 如后续并发和数据量提升，再支持切换到 `MySQL`

### 8.1 主要存储内容

- 故障模板
- 白名单
- 巡检记录
- 定点排查记录
- 大模型配置
- 系统配置

### 8.2 导入导出

系统需要支持 `JSON/YAML` 导入导出，主要用于：

- 模板迁移
- 白名单迁移
- 配置备份

## 9. 部署与架构建议

首版建议采用轻量部署架构：

- `frontend`：前端页面
- `backend`：API、规则引擎、采集调度、大模型调用
- `database`：本地数据库

建议尽量压缩到少量 Pod：

- 前后端可视技术栈情况合并
- 调度任务尽量内聚到后端服务内
- 不单独拆分复杂任务系统

## 10. 非功能需求

### 10.1 安全性

- 系统仅使用 K8s 只读权限
- 系统不执行集群写操作
- 大模型调用只传输排查所需数据
- 大模型结果不直接触发任何动作

### 10.2 性能

- 集群自检单次执行时间应可控
- 命名空间巡检在常规规模下应快速返回结果
- 定点排查匹配响应应尽量控制在秒级

### 10.3 可维护性

- 模板支持页面化维护
- 白名单支持页面化维护
- 数据支持 JSON/YAML 迁移
- 规则和系统配置可持续扩展

## 11. MVP 建议范围

首版建议先做以下最小可用能力：

1. 集群自检：控制面、`containerd`、`kubelet`、`Calico`、`ingress-nginx`、`NVIDIA/Ascend device plugin`
2. 命名空间巡检：`Pod + Service + Ingress + TLS Secret + DaemonSet`
3. 故障模板管理：增删改查、启停、导入导出
4. 白名单管理：按 `namespace + label selector + keyword`
5. 定点排查：支持多 Pod 联合匹配
6. 大模型增强：规则未命中时调用 `Qwen`

## 12. 风险与注意事项

### 12.1 最大风险

最大风险不在采集，而在模板设计质量。

如果模板条件定义过粗，会导致：

- 误报
- 漏报
- 多 Pod 误关联
- 建议不准确

### 12.2 设计注意事项

- 日志关键字必须支持持续优化
- 白名单必须避免影响真实故障识别
- 处理命令必须来自已审核模板，不允许模型自由生成后直接作为标准答案输出
- `label selector` 规范必须与现网部署约定一致，否则对象定位会不稳定

## 13. 结论

本方案适合做成一套轻量、可控、规则驱动的 K8s 故障排查系统。相较于大而全的可观测平台，首版更聚焦实际运维排查效率，落地成本更低，且符合单集群、单管理员、只读排查、不改集群的要求。
