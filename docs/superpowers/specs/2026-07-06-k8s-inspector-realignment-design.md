# K8s Inspector 纠偏与并行实施设计

## 1. 文档目的

本文档用于纠正当前 `K8s Inspector` 的产品方向，并给出后续可并行实施的设计边界。

目标有三点：

- 让系统回到“减少运维排障工作量”的主目标
- 让不同 agent 拿到文档后可以独立推进，不互相影响
- 让后续实现具备清晰的职责边界、统一的数据契约和人性化 UI 原则

本文档是后续多会话并行开发的主参考文档。

## 2. 产品定位纠偏

### 2.1 当前偏差

当前仓库里的骨架设计和页面实现，更偏向一个通用的 K8s 展示系统：

- 强调页面骨架、接口骨架和部署方式
- 巡检入口依然依赖手动输入 `namespace` 和 `label selector`
- 结果页更像资源列表，不像排障工作台

这会带来一个直接问题：如果系统仍然要求用户频繁手动输入、切换页面、自己比对状态和日志，那么它在实际排障效率上并不会明显优于 `Rancher`。

### 2.2 纠偏后的定位

`K8s Inspector` 应定位为一套面向实施运维和平台运维的只读排障系统，而不是通用资源展示平台。

系统核心价值是：

- 减少手动输入
- 减少重复点击和重复检查
- 减少人工比对日志、事件、状态的工作量
- 把常见故障经验沉淀成可以复用、可以匹配的模板

### 2.3 成功标准

如果系统实现正确，用户应能做到：

- 用两个入口快速发起检查：整个名称空间，或单个 Pod
- 在异常 Pod 上直接看到 `status + describe + events + log hits`
- 在结果页直接忽略误报，并自动转成白名单
- 预先录入故障模板后，手动触发检查时无需重新输入范围
- 将常用巡检对象、关键字规则、白名单、故障模板导入导出到其他环境

## 3. 范围定义

### 3.1 本期纳入范围

本期实现聚焦两条主线并行推进：

1. 人用排障主线
2. 故障模板主线

二者并行，但必须共用统一数据契约，避免各自定义一套模型。

### 3.2 本期功能范围

本期系统包含以下能力：

- 名称空间巡检
- 单 Pod 巡检
- 异常 Pod 证据展示
- 日志关键字匹配
- 白名单管理与就地忽略
- 常用巡检对象保存与导入导出
- 故障模板录入、编辑、启停、导入导出
- 按故障模板执行手动检查
- 故障模板命中结果展示

### 3.3 本期明确不纳入范围

- 自动修复
- 自动执行集群命令
- 集群写操作
- 多集群统一纳管
- 复杂权限体系

## 4. 用户工作流

### 4.1 工作流 A：名称空间巡检

用户选择一个名称空间，系统自动检查：

- Pod
- Service
- Ingress
- DaemonSet
- Secret

其中 `Pod` 是核心对象。结果优先展示异常 Pod，并输出：

- Pod 状态
- 重启次数
- 关键事件
- `describe` 摘要
- 日志关键字命中
- 关联错误信息

### 4.2 工作流 B：单 Pod 巡检

用户选择某个 Pod，系统聚焦该 Pod 输出完整证据，不再混入其他对象。

适用场景：

- 已经知道异常 Pod 名称
- 需要快速看一条 Pod 的完整证据链

### 4.3 工作流 C：保存常用巡检对象

用户可保存常用巡检对象，至少包括：

- 对象名称
- `namespace`
- `label selector`
- 默认检查范围

保存后，后续可直接一键巡检，不必重复手动输入。

### 4.4 工作流 D：日志误报忽略

用户在结果页看到某条日志命中后，可以直接点击“忽略此报错”。

该操作等价于新增一条白名单规则。用户不需要再跳到白名单管理页手动录入。

### 4.5 工作流 E：故障模板检查

用户预先录入故障模板。模板内部已经定义：

- 检查对象组
- `namespace`
- `label selector`
- 条件
- 逻辑关系

后续用户手动触发模板检查时，不需要重新指定巡检范围。系统直接按模板定义的目标对象执行检查和匹配。

## 5. 核心功能设计

### 5.1 巡检入口

系统提供三个一级入口：

1. 名称空间巡检
2. 单 Pod 巡检
3. 故障模板检查

入口目标是减少认知负担，不使用“定点排查”这类抽象名称，而是用明确动作命名。

### 5.2 名称空间巡检

名称空间巡检默认检查：

- Pod 状态
- Pod 重启次数
- Pod 最近事件
- Pod `describe` 摘要
- Pod 日志关键字命中
- Service 状态
- Ingress 状态
- DaemonSet 状态
- Secret 状态

处理规则：

- 对象不存在时跳过，不视为异常
- 结果展示时异常优先，正常对象收起或折叠
- `Pod` 结果优先级高于其他对象

### 5.3 单 Pod 巡检

单 Pod 巡检至少输出：

- Pod 基础状态
- 容器状态
- 重启次数
- 关键事件
- `describe` 摘要
- 当前日志命中
- 必要时前一次日志命中

### 5.4 关键字库

系统需要维护自己的关键字库，而不是只依赖用户临时输入关键字。

关键字库分为两部分：

- 系统内置关键字
- 用户维护关键字

关键字至少支持：

- 关键字文本
- 类别
- 严重级别
- 说明
- 启用状态

### 5.5 白名单

白名单用于过滤已知无害的日志命中。

本期白名单基础维度为：

- `namespace + label selector + keyword`

为兼容后续精细化控制，数据模型预留以下可选字段：

- `pod_name_pattern`
- `container_name`

白名单入口分两类：

- 管理页新增/编辑
- 结果页“忽略此报错”直接创建

### 5.6 故障模板

故障模板用于沉淀可量化、可重复匹配的常见故障。

模板内部至少包含：

- 模板名称
- 适用场景
- 检查对象组
- 每个对象组对应的 `namespace + label selector`
- 条件列表
- 条件之间的逻辑关系
- 故障原因
- 处理建议
- 处理命令
- 风险提示
- 启用状态

### 5.7 模板条件

本期模板条件支持以下类型：

- 日志关键字命中
- Pod 状态命中
- 重启次数阈值
- Event 关键字命中
- 关联对象状态异常

模板匹配规则：

- 每个对象组通过 `namespace + label selector` 定位
- 如果某对象组匹配出多个 Pod，则默认“任一 Pod 命中该条件，即该条件成立”

### 5.8 模板执行

手动触发模板检查时：

- 不再要求用户重新输入范围
- 系统直接按模板定义范围采集证据
- 输出命中和未命中的条件
- 给出模板级故障结论

## 6. UI 设计原则

UI 目标不是展示更多信息，而是让用户更快完成排障。

### 6.1 首页是工作台，不是导航页

首页应直接展示：

- 名称空间巡检入口
- 单 Pod 巡检入口
- 故障模板检查入口
- 最近使用的巡检对象
- 最近异常结果

### 6.2 结果页以异常优先

结果页默认优先展示：

- 异常 Pod
- 高风险状态
- 日志关键字命中
- 模板命中结果

不应默认把正常对象和异常对象混在同一层级。

### 6.3 Pod 详情页必须证据集中

用户进入一个 Pod 后，应在尽量少的滚动范围内看到：

- Pod 状态
- 所属节点
- 重启次数
- 事件
- `describe` 摘要
- 日志命中片段
- 忽略此报错操作

### 6.4 模板录入必须可视化

故障模板录入不应要求用户直接编辑原始 JSON。

应提供条件块编辑器，让用户通过界面组合：

- 对象组
- 条件类型
- 运算符
- 目标值
- 条件间逻辑关系

每个条件块都应提供自然语言预览，降低误配置概率。

### 6.5 白名单操作就地完成

结果页上的日志命中必须允许直接：

- 忽略此报错
- 查看已被哪条白名单覆盖

白名单管理页主要用于：

- 批量管理
- 搜索
- 启停
- 导入导出

## 7. 模块拆分、职责与边界

后续实现必须拆成相互独立的模块，严格限制跨层职责。

### 7.1 模块 A：巡检入口与运行编排

职责：

- 接收用户发起的三类检查请求
- 将请求转换成统一巡检目标
- 调用采集层、关键字层、模板层
- 组装统一返回结果

边界：

- 不直接访问 Kubernetes SDK
- 不直接实现模板命中逻辑
- 不直接操作数据库中的白名单规则细节

输入：

- 用户巡检请求
- 已保存巡检对象
- 模板检查请求

输出：

- 统一巡检结果

### 7.2 模块 B：K8s 采集与证据抽取

职责：

- 只读采集 Kubernetes 数据
- 采集 Pod、Event、日志、关联资源状态
- 抽取结构化证据

边界：

- 不做白名单过滤
- 不做模板匹配
- 不负责页面呈现

输入：

- 巡检目标

输出：

- 原始对象数据
- 结构化证据包

### 7.3 模块 C：关键字库与白名单

职责：

- 维护关键字库
- 对日志执行关键字匹配
- 应用白名单过滤
- 处理“忽略此报错”生成白名单规则

边界：

- 不负责采集日志
- 不负责整体模板命中判断
- 不负责页面布局

输入：

- 日志文本
- 关键字规则
- 白名单规则

输出：

- 关键字命中列表
- 白名单过滤结果

### 7.4 模块 D：故障模板与匹配引擎

职责：

- 管理模板定义
- 根据模板条件执行匹配
- 输出模板命中结果和证据引用

边界：

- 不直接拉取 K8s 数据
- 不负责白名单录入页面
- 不负责巡检入口交互

输入：

- 模板定义
- 采集层证据
- 关键字命中结果

输出：

- 模板命中结果
- 命中条件和未命中条件

### 7.5 模块 E：配置与资产管理

职责：

- 管理常用巡检对象
- 管理关键字库导入导出
- 管理白名单导入导出
- 管理模板导入导出

边界：

- 不执行巡检
- 不执行模板匹配
- 不处理运行时证据

输入：

- 配置读写请求

输出：

- 持久化配置资产

### 7.6 模块 F：前端工作台与交互

职责：

- 实现人性化工作台界面
- 承载巡检入口、结果页、模板录入、白名单管理
- 保证操作闭环

边界：

- 不在前端自行实现诊断逻辑
- 不在前端绕过接口直接拼规则

输入：

- 后端统一返回结果

输出：

- 用户可用的工作流界面

## 8. 统一数据契约

为支持多 agent 并行开发，必须先冻结一版统一数据契约。

### 8.1 InspectionTarget

表示一次检查目标。

字段建议：

- `type`: `namespace | pod | template`
- `namespace`
- `pod_name`
- `label_selector`
- `saved_target_id`
- `template_id`
- `resource_scope`

### 8.2 EvidenceBundle

表示一个对象的证据包。

字段建议：

- `object_type`
- `namespace`
- `name`
- `status`
- `node_name`
- `restarts`
- `describe_summary`
- `events`
- `resource_usage`
- `log_hits`
- `related_resources`

### 8.3 KeywordHit

表示一次日志关键字命中。

字段建议：

- `keyword`
- `category`
- `severity`
- `source`
- `matched_text`
- `container_name`
- `whitelisted`
- `whitelist_rule_id`

### 8.4 TemplateCondition

表示模板中的单个条件。

字段建议：

- `target_ref`
- `condition_type`
- `operator`
- `expected_value`
- `join_operator`
- `enabled`

### 8.5 TemplateMatchResult

表示模板级匹配结果。

字段建议：

- `template_id`
- `template_name`
- `matched`
- `matched_conditions`
- `unmatched_conditions`
- `summary`
- `reason`
- `suggestion`
- `risk_note`
- `evidence_refs`

## 9. 并行开发规则

为避免多个 agent 互相影响，必须遵循以下规则：

### 9.1 先冻结契约，再分别开发

优先确定：

- API 请求响应结构
- 数据模型字段
- 前后端命名

在契约冻结前，不允许各模块自行扩散字段定义。

### 9.2 禁止跨模块直接依赖内部实现

模块之间只能通过：

- API
- service interface
- schema

进行交互，不能直接引用对方内部细节。

### 9.3 白名单与模板必须解耦

白名单只负责过滤日志命中，不能直接决定模板是否存在。

模板引擎只消费“过滤后的命中结果”和“原始证据引用”，不应反向控制白名单规则。

### 9.4 UI 与诊断逻辑必须解耦

前端页面不维护独立的匹配逻辑。

UI 只负责：

- 触发动作
- 展示结果
- 提交配置

### 9.5 采集层与展示层必须解耦

采集层返回标准化证据对象，不按具体页面拼装返回结构。

否则后续新增入口会反复重做采集逻辑。

### 9.6 并行开发的文件边界

为减少多个 agent 修改同一文件，建议按目录边界切分：

- 任务 1 优先修改：
  - `backend/app/schemas/`
  - `backend/app/models/`
  - `backend/app/api/routes/` 中的契约相关接口定义
- 任务 2 优先修改：
  - `backend/app/services/inspection_service.py`
  - `backend/app/api/routes/inspections.py`
  - 新增巡检编排相关 service 文件
- 任务 3 优先修改：
  - `backend/app/providers/`
  - `backend/app/services/` 中与采集和证据抽取相关文件
- 任务 4 优先修改：
  - `backend/app/services/whitelist_service.py`
  - 新增关键字库 service
  - `backend/app/api/routes/whitelists.py`
- 任务 5 优先修改：
  - `backend/app/engine/`
  - `backend/app/services/template_service.py`
  - `backend/app/services/diagnosis_service.py`
  - `backend/app/api/routes/templates.py`
  - `backend/app/api/routes/diagnoses.py`
- 任务 6 优先修改：
  - `frontend/src/pages/`
  - `frontend/src/features/`
  - `frontend/src/components/`
  - `frontend/src/styles.css`

限制规则：

- 非本任务负责目录只允许为接接口做最小改动
- 共享文件改动前，必须先确认统一契约已经冻结
- `backend/app/schemas/` 和 `frontend/src/api/types.ts` 属于高冲突文件，必须优先统一后再扩展

## 10. 建议实施顺序

虽然后续可以多会话并行，但建议按以下顺序推进：

1. 统一数据契约
2. 巡检入口与后端运行编排
3. K8s 采集与证据抽取
4. 关键字库与白名单
5. 故障模板与匹配引擎
6. 配置与资产管理
7. 前端工作台与人性化 UI

说明：

- 第 1 步是所有并行任务的共同前提
- 第 2 到第 6 步可以部分并行
- 第 7 步需要在接口基本稳定后集中收口

## 11. Agent 任务拆分

后续建议拆成以下 6 个任务，分别给不同 agent 执行。

### 任务 1：统一契约与数据模型

目标：

- 定义统一 schema
- 对齐数据库模型与 API 模型
- 为其他任务提供稳定边界

产出：

- 后端 schema
- 数据模型说明
- API 契约文档

不负责：

- 页面开发
- K8s 实际采集

### 任务 2：巡检编排与检查入口

目标：

- 建立名称空间巡检、单 Pod 巡检、模板检查的统一入口
- 组装运行流程

产出：

- 后端巡检入口服务
- 巡检请求与返回结构

不负责：

- 关键字匹配实现细节
- 模板条件引擎内部实现

### 任务 3：K8s 采集与证据抽取

目标：

- 实现只读采集
- 输出结构化证据包

产出：

- provider 采集逻辑
- Pod/Event/Log/关联资源证据抽取

不负责：

- UI
- 白名单逻辑

### 任务 4：关键字库与白名单

目标：

- 实现关键字库管理
- 实现日志命中和白名单过滤
- 实现“忽略此报错”

产出：

- 关键字匹配服务
- 白名单服务
- 导入导出能力

不负责：

- 模板组合匹配
- K8s 采集

### 任务 5：故障模板与匹配引擎

目标：

- 实现模板录入、编辑、启停
- 实现条件匹配和模板命中结果

产出：

- 模板服务
- 匹配引擎
- 模板执行结果结构

不负责：

- 白名单规则录入
- 页面整体视觉设计

### 任务 6：前端工作台与人性化 UI

目标：

- 把以上能力做成可用工作流
- 保证低输入、高可读、异常优先、就地操作

产出：

- 工作台首页
- 名称空间巡检页
- 单 Pod 详情页
- 模板录入页
- 白名单管理页

不负责：

- 后端规则判断
- 采集实现

## 12.5 任务依赖关系

不同任务可以并行，但依赖关系必须明确：

- 任务 1 是所有任务的前置条件
- 任务 2、任务 3、任务 4、任务 5 在任务 1 完成后可并行
- 任务 6 可提前做低保真页面结构，但正式联调应在任务 1 和任务 2 的接口稳定后进行
- 如果任务 4 改动 `KeywordHit`，必须同步回到任务 1 统一契约，不允许私自扩字段
- 如果任务 5 增加新的模板条件类型，必须同步更新任务 1 的 `TemplateCondition` 定义

## 13. UI 验收重点

前端交付时必须重点验证以下事项：

- 首页是否一眼能看懂要做什么
- 巡检是否尽量复用保存对象，减少重复输入
- 结果页是否异常优先而不是全量平铺
- Pod 详情是否能集中展示证据
- 日志命中是否支持就地忽略
- 模板录入是否能让非开发人员理解条件含义

如果这些点没有满足，即使功能齐全，也不算达到本期目标。

## 14. 结论

后续实现必须围绕“减少排障工作量”展开，而不是继续扩成通用资源展示系统。

本期最重要的工程原则有三条：

- 统一契约先行
- 模块边界清晰
- UI 以排障效率为中心

后续所有任务都应以本文档为主约束，不再以“页面先搭起来”为优先目标。
