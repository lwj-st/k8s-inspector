# K8s Inspector 自动巡检产品纠偏与下一轮开发指令

## 1. 本文档结论

本文档用于纠正前一轮产品方向偏差，并作为下一轮多个开发会话的主参考。

## 0. 用户已确认的产品决策

确认时间：2026-07-12。

用户已确认以下产品方向：

1. 名称空间巡检默认展示全部名称空间，并支持多选与一键巡检全部。
2. 名称空间巡检主流程不需要导入导出巡检对象。
3. 自动巡检页可以作为新的主首页，替代当前手动名称空间巡检页。
4. 模板检查只按模板内配置范围执行，不允许用户在检查页临时输入 namespace。

额外 UI 原则：

1. 后续开发必须按功能逐步交付，不要一次性铺满所有能力。
2. 每个功能完成后必须能让用户真实使用，而不是只完成接口或只完成页面。
3. 导入导出这类低频操作不能在主页面长期占大块空间，应该放到弹窗、抽屉、更多菜单或独立管理页。
4. 页面布局要先保证用户路径清晰，再考虑能力完整性。

前一轮实现把“名称空间巡检”做成了偏手工的流程：用户输入名称空间、保存巡检对象、导入导出巡检对象。这不符合目标用户的使用习惯，也没有明显降低运维排障成本。

新的产品主线必须调整为：

1. 系统自动发现集群内所有名称空间。
2. 用户默认从名称空间列表中选择或批量选择，不手动输入名称空间。
3. 名称空间巡检必须检查 Pod 状态、容器状态、重启、事件、describe 摘要、日志关键字命中、关联对象异常，而不是只查 Pod 状态。
4. 名称空间巡检主流程不需要导入导出“巡检对象”。
5. 导入导出只保留给跨环境迁移的配置类数据：故障模板、关键字库、白名单。是否保留“保存目标”能力需弱化为高级功能，不能放在主流程。
6. 故障模板用于沉淀人工录入的可量化故障经验。模板已绑定名称空间、label、对象组和条件，手动触发模板检查时系统自动按模板采集和匹配。

## 2. 已确认的产品需求

以下需求已由用户确认，后续所有 agent 按第 5 节任务指令执行。

1. 名称空间发现：
   - 系统进入页面后自动读取当前集群所有名称空间。
   - 用户不需要手动输入名称空间。
   - UI 支持搜索、勾选多个名称空间、全选/反选。

2. 名称空间巡检触发：
   - 支持“巡检选中的名称空间”。
   - 支持“一键巡检全部名称空间”。
   - 每个名称空间独立展示巡检状态、异常数量和最近巡检时间。

3. 名称空间巡检内容：
   - Pod phase、container waiting/terminated reason、ready 状态。
   - restart count、CrashLoopBackOff、ImagePullBackOff、OOMKilled、Evicted、Pending、Unknown 等常见状态。
   - Event 关键字和 Warning event。
   - describe 摘要。
   - 当前日志和 previous 日志的关键字命中。
   - Service、Ingress、DaemonSet、Secret/TLS 等关联对象异常。
   - 日志命中需要走系统关键字库和白名单过滤。

4. 页面主流程：
   - 名称空间巡检页不展示导入导出巡检对象。
   - 名称空间巡检页不把“保存常用范围”作为主操作。
   - 主页面应是：集群概览 -> 名称空间列表 -> 异常优先 -> 证据抽屉/详情页。

5. 故障模板：
   - 模板录入时可以指定名称空间、label、对象组。
   - 如果 label 匹配多个 Pod，任一 Pod 满足该对象组条件即视为该对象组命中。
   - 模板检查入口不要求用户重复输入名称空间。
   - 模板检查结果必须展示命中条件、未命中条件、证据来源、故障原因和建议。

6. 手动 Pod 巡检：
   - 仍保留，但不是主入口。
   - Pod 名称最好从已发现的名称空间和 Pod 列表中选择，允许高级用户手动输入。

## 3. 当前实现与目标的偏差

### 3.1 名称空间输入方式错误

当前 UI 仍要求用户手动输入名称空间。这和“减少工作量”的目标冲突。

正确方向：

- 后端提供名称空间发现接口。
- 前端自动加载名称空间列表。
- 用户通过搜索和勾选选择目标。

### 3.2 名称空间巡检主流程错误

当前页面把“保存巡检对象、导入、导出”放在主流程里，导致页面长、认知负担重。

正确方向：

- 名称空间巡检主流程只保留发现、选择、巡检、看异常、看证据。
- 导入导出不属于名称空间巡检主流程。
- 保存常用范围如果保留，只能放到高级设置或二级入口。

### 3.3 巡检结果信息密度不合理

当前页面按资源列表堆叠信息，用户需要滚动很长页面才能定位问题。

正确方向：

- 默认只突出异常。
- 正常对象折叠。
- 名称空间列表展示摘要。
- 点击异常名称空间或 Pod 后进入右侧详情抽屉。
- 详情抽屉展示 describe、event、log hit、白名单操作、关联对象。

### 3.4 产品文档需要废弃部分旧假设

旧文档中关于“常用巡检对象保存与导入导出”的设计不应再作为名称空间巡检主流程依据。

后续 agent 应以本文档为主，不再按旧文档继续强化手动保存巡检对象流程。

## 4. 新目标架构

### 4.1 后端能力分层

1. 集群发现层：
   - 发现名称空间。
   - 发现指定名称空间下的 Pod。
   - 提供轻量摘要，供 UI 快速渲染选择器。

2. 巡检编排层：
   - 执行单名称空间巡检。
   - 执行多名称空间批量巡检。
   - 执行单 Pod 巡检。
   - 执行故障模板检查。
   - 管理巡检任务状态和历史记录。

3. K8s 证据采集层：
   - 从 Kubernetes API 采集 Pod、Event、Service、Ingress、DaemonSet、Secret/TLS。
   - 采集当前日志和 previous 日志。
   - 产出原始证据，不负责 UI 展示结构。

4. 证据分析层：
   - 识别异常状态。
   - 匹配关键字库。
   - 应用白名单过滤。
   - 生成统一 `EvidenceBundle` 和 `KeywordHit`。

5. 模板匹配层：
   - 只消费统一证据和模板定义。
   - 不直接调用 Kubernetes。
   - 不绕过关键字库和白名单。

### 4.2 前端信息架构

前端主导航建议调整为：

1. 自动巡检
2. 故障模板
3. 关键字与白名单
4. 系统配置

“自动巡检”页面内包含：

1. 顶部集群状态摘要。
2. 名称空间搜索与筛选。
3. 名称空间列表，展示异常数、Pod 数、最近巡检时间、状态。
4. 操作按钮：巡检选中、巡检全部、只看异常。
5. 巡检结果区：按名称空间分组，异常优先。
6. 右侧详情抽屉：Pod 证据、日志命中、describe、event、忽略误报。

手动 Pod 巡检可以作为“自动巡检”里的高级入口，不应成为主导航最显眼入口。

## 5. 下一轮开发指令

### 5.1 给“统一契约与数据模型”的指令

让“统一契约与数据模型” agent 参考本文档，先做契约纠偏，不做 UI。

必须完成：

1. 新增名称空间发现契约：
   - `NamespaceSummary`
   - `NamespaceDiscoveryResponse`
   - 字段至少包含 `name`、`status`、`pod_count`、`abnormal_pod_count`、`last_inspected_at`、`labels`。

2. 新增批量巡检契约：
   - `NamespaceInspectionRequest` 支持单个名称空间。
   - 新增 `NamespaceBatchInspectionRequest` 支持多个名称空间或全部名称空间。
   - 新增 `NamespaceBatchInspectionResponse`，按名称空间返回巡检摘要和详情引用。

3. 明确异常分类枚举：
   - Pod 状态异常。
   - 容器状态异常。
   - Event 异常。
   - 日志关键字异常。
   - 关联对象异常。

4. 明确 `KeywordHit` 仍是日志命中统一结构，模板日志条件只能消费 `KeywordHit`。

5. 明确名称空间巡检主流程不再依赖 `SavedInspectionTarget`。

禁止事项：

- 不新增页面。
- 不实现 Kubernetes 采集。
- 不继续扩展巡检对象导入导出作为主流程。

验收标准：

- 后端 schema 与 `frontend/src/api/types.ts` 对齐。
- 契约文档更新清楚。
- 相关 schema 测试通过。

### 5.2 给“巡检编排与检查入口”的指令

让“巡检编排与检查入口” agent 参考本文档和“统一契约与数据模型”产出的契约，重做巡检入口设计。

必须完成：

1. 新增名称空间发现接口：
   - 建议：`GET /api/v1/discovery/namespaces`
   - 返回集群内全部名称空间及轻量摘要。

2. 新增批量名称空间巡检接口：
   - 建议：`POST /api/v1/inspections/namespaces/run`
   - 支持传入多个 namespace。
   - 支持 `all_namespaces: true`。

3. 编排层负责：
   - 批量拆分名称空间。
   - 聚合每个名称空间巡检状态。
   - 记录历史。
   - 返回异常优先的结果。

4. 编排层不负责：
   - 直接解析日志关键字。
   - 直接拼 UI 文案。
   - 直接执行模板匹配。

5. 对旧的 `SavedInspectionTarget`：
   - 不删除已有代码。
   - 从名称空间主流程中降级为兼容/高级能力。
   - 不再新增围绕保存目标的主流程能力。

验收标准：

- 支持自动发现名称空间。
- 支持一键巡检全部名称空间。
- 支持选中多个名称空间巡检。
- 后端测试覆盖空集群、单 namespace、多 namespace、全部 namespace。

### 5.3 给“K8s 采集与证据抽取”的指令

让“K8s 采集与证据抽取” agent 参考本文档，补齐名称空间巡检的证据质量。

必须完成：

1. 名称空间发现：
   - 基于 Kubernetes API list namespace。
   - 不依赖用户手工输入。

2. Pod 证据：
   - phase。
   - ready 状态。
   - container waiting reason。
   - container terminated reason。
   - restart count。
   - node name。
   - image pull 相关错误。
   - OOMKilled、Evicted、CrashLoopBackOff、Pending、Unknown。

3. Event 证据：
   - Warning event。
   - BackOff、Failed、Unhealthy、FailedScheduling、FailedMount、FailedPull、ImagePullBackOff 等常见关键字。

4. 日志证据：
   - 当前日志。
   - previous 日志。
   - 按关键字库匹配。
   - 命中后输出 `KeywordHit`。

5. 关联对象证据：
   - Service endpoints 异常。
   - Ingress backend/TLS 异常。
   - DaemonSet desired/current/ready 不一致。
   - Secret/TLS 缺失或格式异常。

禁止事项：

- 不做模板匹配。
- 不做前端页面。
- 不绕过关键字库服务硬编码大量业务规则。内置关键字应进入关键字库。

验收标准：

- provider 测试覆盖典型异常状态。
- 日志和 previous 日志命中可区分来源。
- 正常对象不会被误报为异常。

### 5.4 给“关键字库与白名单”的指令

让“关键字库与白名单” agent 参考本文档，围绕自动巡检修正边界。

必须完成：

1. 扩充内置关键字库：
   - connection refused。
   - timeout。
   - OOMKilled。
   - CrashLoopBackOff。
   - ImagePullBackOff。
   - ErrImagePull。
   - permission denied。
   - no such host。
   - failed to connect。
   - readiness probe failed。
   - liveness probe failed。

2. 明确白名单匹配优先级：
   - namespace。
   - label selector。
   - pod name pattern。
   - container name。
   - keyword。

3. 白名单必须影响：
   - 自动名称空间巡检结果。
   - 单 Pod 巡检结果。
   - 故障模板匹配结果。

4. “忽略此报错”创建白名单后，当前结果应立即标记为已忽略。

禁止事项：

- 不做 Kubernetes 采集。
- 不做模板条件匹配。
- 不把白名单做成只影响前端本地状态。

验收标准：

- 白名单过滤在后端生效。
- 模板匹配不会命中已白名单过滤的日志异常。
- 有测试覆盖 pod_name_pattern 和 container_name。

### 5.5 给“故障模板与匹配引擎”的指令

让“故障模板与匹配引擎” agent 参考本文档，保持模板作为“故障经验匹配器”，不要变成手动巡检表单。

必须完成：

1. 模板录入仍允许指定：
   - namespace。
   - label selector。
   - 对象组。
   - 条件。
   - 逻辑关系。

2. 模板检查时：
   - 不要求用户输入 namespace。
   - 根据模板目标自动触发采集或复用最新巡检证据。
   - 多个 Pod 匹配同一对象组时，任一 Pod 满足条件即可。

3. 模板条件必须消费统一证据：
   - `KeywordHit`。
   - Pod 状态异常。
   - Event 异常。
   - restart count。
   - 关联对象异常。

4. 结果必须输出：
   - 命中模板。
   - 未命中模板。
   - 命中条件。
   - 未命中条件。
   - 证据来源。
   - 故障原因。
   - 建议动作。

禁止事项：

- 不绕过白名单扫描原始 log_summary。
- 不要求用户在模板检查页重新输入巡检范围。
- 不把模板导入导出放进名称空间巡检页。

验收标准：

- 复杂模板测试覆盖 pod1 日志 + pod2 日志 + 某 Pod 状态 fail + 其他条件。
- label 匹配多个 Pod 时，任一 Pod 满足即成立。
- 白名单日志不会触发模板命中。

### 5.6 给“前端工作台与人性化 UI”的指令

让“前端工作台与人性化 UI” agent 参考本文档，重新设计主体验，不要继续修补当前长页面。

必须完成：

1. 主导航调整：
   - 自动巡检。
   - 故障模板。
   - 关键字与白名单。
   - 系统配置。

2. 自动巡检页：
   - 自动加载名称空间列表。
   - 支持搜索名称空间。
   - 支持多选。
   - 支持巡检选中。
   - 支持巡检全部。
   - 支持只看异常。
   - 展示每个名称空间的摘要卡片或表格行。

3. 结果交互：
   - 默认异常优先。
   - 正常对象折叠。
   - 点击异常 Pod 打开详情抽屉。
   - 抽屉内展示状态、容器、event、describe、日志命中、忽略按钮。

4. 删除或下沉：
   - 名称空间巡检页主流程中的保存巡检对象。
   - 名称空间巡检页主流程中的导入导出。
   - 大段说明性文字。

5. UI 原则：
   - 页面不能靠纵向长滚动堆信息。
   - 每屏只解决一个问题：选范围、看摘要、看详情。
   - 主按钮不能过大。
   - 文案必须贴近运维动作，不使用抽象英文标题。
   - 空状态要告诉用户下一步做什么。

6. 低频操作放置规则：
   - 导入导出只能出现在故障模板、关键字库、白名单这类配置管理场景。
   - 导入导出入口必须放在“更多操作”、弹窗、抽屉或独立管理页。
   - 禁止在巡检主页面常驻展示导入 textarea、导出 textarea 或大块 JSON 区域。
   - 保存常用范围如果后续仍保留，只能作为高级能力，不能出现在自动巡检首屏主流程。
   - 名称空间巡检主页面不得出现“保存巡检对象导入/导出”。

7. 布局验收规则：
   - 首屏必须能看到名称空间列表和主要巡检按钮。
   - 用户不滚动也能理解下一步动作。
   - 异常结果必须比正常结果更醒目。
   - 详情信息使用右侧抽屉、弹窗或分栏，不允许把所有 describe/log/event 平铺成长页面。
   - 表单字段必须有明确业务含义，例如“名称空间”“保存为”“Label Selector（可选）”，禁止让用户猜字段用途。

禁止事项：

- 不私自实现后端没有的 mock 业务。
- 不把导入导出继续放在自动巡检主界面。
- 不让用户手动输入 namespace 作为默认路径。

验收标准：

- 用户进入自动巡检页后，不输入任何 namespace，也能看到集群名称空间列表。
- 用户可以一键巡检全部名称空间。
- 巡检结果能快速定位异常 Pod 和日志命中。
- 页面首屏可理解，不依赖阅读大段说明。

## 6. 推荐执行顺序

### 6.0 垂直切片交付原则

后续不要按“后端全做完、前端再全部重做”的方式堆大版本。应按功能切片交付，每个切片都必须包含契约、后端、前端、测试和最小可用验收。

推荐切片顺序：

1. 切片一：自动发现名称空间。
   - 用户进入系统即可看到名称空间列表。
   - 不需要手动输入 namespace。
   - 不涉及日志、不涉及模板、不涉及复杂详情。

2. 切片二：选中名称空间并触发巡检。
   - 支持多选。
   - 支持巡检选中。
   - 支持巡检全部。
   - 每个名称空间展示巡检中、成功、失败、异常数量。

3. 切片三：名称空间巡检异常摘要。
   - 先覆盖 Pod 状态、重启、容器异常、Warning event。
   - 异常优先展示。
   - 正常项折叠。

4. 切片四：Pod 证据详情。
   - 点击异常 Pod 打开详情抽屉。
   - 展示 describe 摘要、events、当前日志命中、previous 日志命中。
   - 支持“忽略此报错”并落白名单。

5. 切片五：模板检查接入自动证据。
   - 模板检查只按模板内配置范围执行。
   - 输出命中/未命中条件和证据来源。
   - 不允许检查页临时输入 namespace。

6. 切片六：UI 收口。
   - 清理主流程里的低频操作。
   - 导入导出统一放弹窗、抽屉或更多菜单。
   - 修正页面过长、按钮过大、标题不清晰等问题。

每个切片完成后必须留下 worklog，说明：

- 完成了哪个用户动作。
- 用户现在怎么用。
- 哪些能力刻意没做。
- 跑了哪些测试。
- 是否影响其它模块。

### 6.1 串行前置

第一步只让“统一契约与数据模型” agent 开工。

原因：

- 名称空间发现、批量巡检、异常摘要都需要新契约。
- 契约未定时，其它 agent 容易继续扩散旧模型。

### 6.2 后端并行

“统一契约与数据模型”完成并留下 worklog 后，可并行启动：

1. “巡检编排与检查入口”
2. “K8s 采集与证据抽取”
3. “关键字库与白名单”
4. “故障模板与匹配引擎”

并行限制：

- 任何契约变更必须回到“统一契约与数据模型”。
- 模板引擎不得直接调用 Kubernetes。
- 采集层不得实现模板匹配。
- 白名单必须在后端统一生效。

### 6.3 前端后置

“前端工作台与人性化 UI”可以先做静态交互稿，但正式联调必须等以下任务完成：

1. 名称空间发现接口。
2. 批量名称空间巡检接口。
3. 统一异常摘要契约。
4. 证据详情契约。

### 6.4 最终验收

最后由总控或验收会话执行：

1. 后端全量测试。
2. 前端全量测试。
3. 生产构建。
4. 使用真实 kubeconfig 在开发集群验证：
   - 自动发现名称空间。
   - 巡检全部名称空间。
   - 查看异常 Pod 证据。
   - 忽略日志误报。
   - 运行故障模板检查。

## 7. 给用户的确认问题

请用户确认以下 4 点后再安排 agent 开发：

1. 名称空间巡检是否默认展示全部名称空间，并支持多选与一键巡检全部？
2. “保存巡检对象/导入导出巡检对象”是否从名称空间巡检主流程移除，仅保留为高级能力或直接废弃？
3. 自动巡检页是否作为新的主首页，替代现在的手动名称空间巡检页？
4. 模板检查是否只按模板内配置范围执行，不再允许用户在检查页临时输入 namespace？
